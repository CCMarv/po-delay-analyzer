El Problema

Por que existe este proyecto?

Los equipos de supply chain revisan cientos de Purchase Orders manualmente buscando retrasos y sus causas. El proceso es lento, inconsistente y propenso a errores humanos.

  • Un PO puede retrasarse por 5+ razones diferentes (vendor, carrier, DC, scheduling)
  • La causa real esta escondida en multiples timestamps del lifecycle
  • El DC staff ingresa reason codes manualmente — a veces no coinciden con lo que paso
  • Nadie genera una explicacion clara y accionable para cada caso
  • El equipo necesita priorizar: cuales POs arreglar primero?

06
Logica de Clasificacion

Reglas de delay por etapa

Stage 1: Pre-Arrival (Vendor / Carrier)
STA Push: APPROVED_DT > STA_DT Late Shipment: VENDOR_SHIP_DT > STA_DT Carrier Miss: TRAILER_ARRIVE > APPROVED_DT + threshold Rescheduled: CURRENT_APPROVED != FIRST_APPROVED
Responsable: Vendor o Carrier segun campo que exceda umbral
Stage 2: At the DC (DC Controlled)
Yard Congestion: CHECKIN_DT - TRAILER_ARRIVE > 4 hrs Long Dock: CHECKOUT_DT - CHECKIN_DT > 6 hrs
Responsable: DC Operations — el trailer ya llego pero la recepcion se retraso
Cross-Stage / Prioridad
Late Receipt: RECPT_DT > STA_DT (KPI principal) Short Ship: CASES_SHIPPED / CASES_ORDERED < 0.9 Hot PO Delayed: HOT_PO_FLAG = 1 + any delay Short Lead Time: STA_DT - PO_DT < 3 days
Estos se superponen — un PO puede tener multiple causas

El valor del REASON_CODE

REASON_DSC es la anotacion humana del staff del DC. El equipo puede:

  • Computar la etapa de delay con timestamps (logica)
  • Comparar con el reason code humano
  • Matches = validacion de que la regla funciona
  • Mismatches = donde el AI agrega valor real sobre la anotacion manual
  • Usar ambos como few-shot examples para el LLM
Tip: El KPI principal es RECPT_DT > STA_DT. Si el receipt fue a tiempo, el PO no tiene delay — independientemente de lo que paso en el camino.
07
Prompt Engineering

Como estructurar el prompt del LLM

Template de referencia
### System You are a supply chain analyst that investigates Purchase Order delays. You explain root causes clearly and recommend specific actions. ### User Analyze this delayed PO: PO: {PO_NBR} | Vendor: {VENDOR_NAME} DC: {DC_ID} | Carrier: {CARRIER_NAME} Timeline: PO Placed: {PO_DT} STA (promised): {STA_DT} Appt Approved: {APPROVED_DT} Trailer Arrived: {TRAILER_ARRIVE_DT} Check-In: {CHECKIN_DT} Check-Out: {CHECKOUT_DT} Receipt Posted: {RECPT_DT} Computed delays: STA to Arrival: {days_late} days late Yard wait: {yard_hours} hrs Dock time: {dock_hours} hrs Classified stage: {STAGE} DC Reason Code: {REASON_DSC} Hot PO: {HOT_PO_FLAG} Short Ship: {short_ship_pct}% Generate: (1) root cause explanation in 2-3 sentences, (2) recommended action, (3) severity: HIGH/MEDIUM/LOW

Que debe incluir el output

  • Identificar la etapa exacta: vendor, carrier, o DC
  • Cuantificar el delay (X dias / Y horas)
  • Explicar si el reason code del DC coincide o no
  • Mencionar si es hot PO o short ship (agravantes)
  • Accion concreta: contactar vendor, escalar a carrier, revisar capacidad DC
  • Severidad: HIGH si hot PO + delay, MEDIUM si delay normal, LOW si borderline
El REASON_DSC no siempre es correcto. Es ingresado manualmente por DC staff y puede estar equivocado. El LLM debe usar los timestamps como fuente de verdad y mencionar discrepancias con el reason code.
