# Hybrid Severity: the LLM emits it, the Phase 2 rule audits it

* **Status:** 🟢 **CURRENT** (closed 2026-06-27)
* **Technical Context:** Phase 3 / LLM Integration — source of severity in the deliverable output
* **References:** Issue #92, #93, #97, #98; kickoff (`documentation/kickoff_po_root_cause.html`, sections 03 and 08); README mentor §6 (Severity Ranking >95%); `02_clasif_reglas_negocio/classifier_core.py` (`_severidad`); ADR-01 (deterministic principle)

## Context and Problem
The deliverable output reports a severity HIGH/MEDIUM/LOW per delayed PO. Two sources of truth from the project place it in different spots, and the scaffolding code of Phase 3 incorrectly merged them:

1. The **kickoff** (section 03, mentor's prompt template) states severity as **LLM output**: *"Generate: (1) root cause explanation, (2) recommended action, (3) severity: HIGH/MEDIUM/LOW"*. It is one of the three products generated by the model, along with explanation and action.
2. The **Severity Ranking metric** (kickoff section 08 / README §6) defines it with a **deterministic rule and a quantitative target**: *"hot POs with delay > 3 days must have severity=HIGH in >95% of cases"*. This is only evaluable if the relationship is reproducible.

The initial scaffolding left the LLM assigning severity with a rule written in the prompt whose threshold (*"delay > 7 days"*) **does not correspond** to either the kickoff or the README (both set **> 3 days**). Meanwhile, Phase 2 is already calculating a deterministic `severity` column with the correct threshold. Two conflicting severities and an incorrect threshold in the prompt remained.

## Considered Options

### Option A: The deterministic severity of Phase 2 is the official one; the LLM only explains it
* **Pros:** Auditable and reproducible; meets the >95% by design.
* **Cons:** **Deviates from the kickoff**, which explicitly requests severity as LLM output. It voids the metric (it is fulfilled by design, nothing is validated).

### Option B: The LLM emits severity and is measured against the rule
* **Pros:** Faithful to the kickoff. The >95% metric becomes a real validation (does the LLM respect hot+delay>3d⇒HIGH?).
* **Cons:** The deliverable's severity is not 100% reproducible; it is necessary to measure and report the %.

### Option C (hybrid): The LLM emits the official severity; Phase 2 rule audits it
* **Pros:** Honors both sources. The LLM generates severity (kickoff); the deterministic rule of Phase 2 remains as a control column that feeds the Severity Ranking metric and the LLM-vs-rule matching analysis. It converts the tension between sources into a measurable finding.
* **Cons:** It maintains two columns and requires defining which is the official one and how to report the discrepancy.

## Decision
The **Option C (hybrid)** was chosen.

1. The **official** severity column of the CSV deliverable (#97) is the one **emitted by the LLM** (`llm_severidad` → `severity` in `po_output.csv`), faithful to the kickoff that defines it as model output.
2. The **deterministic rule of Phase 2** (`severity`, `flag_hot_late & delay_days_calc > 3.0`) is retained as an **audit column**: it feeds the Severity Ranking metric >95% (#98) and allows reporting where the LLM disagrees with the rule.
3. The **prompt threshold is corrected** from the erroneous *"> 7 days"* to the official guideline: **hot PO + delay > 3 days ⇒ HIGH** (#93). This correction is unequivocal in both sources and applies independently of the chosen option.

## Consequences
* **Positive:** The deliverable meets the kickoff (severity as LLM output) without losing auditability: the Phase 2 rule remains present as control and provides real content to the >95% metric (measures if the LLM respects the rule, not taking it for granted). The LLM-vs-rule discrepancy becomes a reportable finding, aligned with the project's spirit ("where AI coincides or not with the reference").
* **Negative:** The output drags two severity columns with distinct roles, requiring clear documentation of which is the official one and which is the auditor to avoid confusing downstream data consumers (#97, #100). If the LLM violates the rule in too many cases, the >95% will expose it, necessitating either iterating the prompt (#93) or reporting it as a limitation of the model.

## Closure (2026-06-27)
Verified that the implementation materializes the decision: the CSV deliverable maps `severity ← llm_severidad` (`03_llm_integration/llm_integration.py`, `export_deliverable_csv`); the deterministic rule of Phase 2 (`02_clasif_reglas_negocio/classifier_core.py`, `_severidad`, `flag_hot_late & delay_days_calc > 3.0`) is retained as an audit column outside the deliverable; and the prompt threshold uses `> 3 days`, according to the kickoff (§08) and the README (§6), not the `> 7 days` from the initial scaffolding. With all three commitments met, the decision moves from Draft to Current.
