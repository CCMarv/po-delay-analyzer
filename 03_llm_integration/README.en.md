# 03 — Integration with LLM

Generates root cause explanations for delayed Purchase Orders (POs), using an LLM to analyze the metrics calculated in previous phases and produce a structured diagnosis.

## What it does

It takes the already cleaned and classified DataFrame (phases 1 and 2), filters the delayed POs (`delay_days_calc > 0`), and for each one constructs a prompt with the complete context (dates, yard/dock metrics, automatic classification, reason code from the DC) that is sent to an LLM. The response is parsed to JSON and these columns are added to the DataFrame:

| Column | Description |
|---|---|
| `llm_root_cause` | 1-2 line explanation generated by the model |
| `llm_recommended_action` | Specific suggested action, with responsible party |
| `llm_severity` | `HIGH`, `MEDIUM`, or `LOW` |
| `llm_matches_reason` | `True`/`False` if the cause matches the reason code from the DC |
| `llm_confidence` | Score from 0.0 to 1.0 |

## Supported backends

| Backend | Engine | Requires API key | Cost |
|---|---|---|---|
| `openai` | OpenAI API (`gpt-4o-mini`) | Yes | Paid — **official backend for the deliverable** |
| `local` | Qwen 2.5:7B via Ollama | No | Free (runs on your machine) |
| `claude` | Claude (Anthropic API) | Yes | Paid |
| `deepseek` | DeepSeek API | Yes | Paid |

The `po_output.csv` of the deliverable is generated with `--backend openai`; the others are alternatives (local development with `local`, or comparison with `claude`/`deepseek`).

## API key configuration

Keys **should never** be written in the code or passed as arguments in production. They are stored in a `.env` file at the root of the repo (same level as `pyproject.toml`), which is excluded from git via `.gitignore`.

1. Copy the template (from the root of the repo):
   ```bash
   cp .env.example .env
   ```
2. Edit `.env` with your real keys (only those of the backend you will use):
   ```
   ANTHROPIC_API_KEY=sk-ant-...
   OPENAI_API_KEY=sk-...
   DEEPSEEK_API_KEY=sk-...
   ```
3. Ensure that `.env` is in `.gitignore`:
   ```bash
   grep "^\.env$" .gitignore
   ```

The script loads these variables automatically with `load_dotenv()`, so there is no need to pass `--api-key` in the terminal once the `.env` is configured.

Each person uses their **own** key (they are not shared; the variable name is standard per provider). In case of suspected leakage, rotate the key immediately in the provider's panel and notify the team. The details of the secret policy are in [ADR-11](../documentation/decisiones/ARD-11.en.md).

## Output

The run produces two artifacts in `../data/processed/`:

**1. Internal Artifact** — the complete DataFrame with all columns (those from F1/F2 plus the `llm_*`), named according to mode and backend:

- `df_with_llm_test_{backend}.csv`
- `df_with_llm_full_{backend}.csv`
- `df_with_llm_{limit}_{backend}.csv`

This is the working and auditing input (includes, e.g., the deterministic `severity` from F2 alongside `llm_severity`). Partial saves every 50 POs (configurable via `DEFAULT_SAVE_EVERY`) are also generated to avoid losing progress in long runs.

**2. Deliverable CSV** — `po_output.csv`, the artifact of the **contract F3→F4** (#100): the only input for Phase 4. It has **33 columns** in four blocks, in this order: the base contract (16, without tier numbering) and tier-1/tier-2 extensions established by [ARD-21](../documentation/decisiones/ARD-21.en.md). The verified operational copy 33/33 resides in [`04_app/README.md`](../04_app/README.en.md#1-entry-the-contract-f3f4-100); this section explains the origin, ARD-21 is the design record.

**Block 1 — Mentor's contract (the 5 columns it evaluates, first and in order):**

| Column | Source | Note |
|---|---|---|
| `PO_NBR` | `PO_NBR` | identity |
| `stage` | `stage_primary` (F2) | Vendor / Carrier / DC / Indeterminate |
| `severity` | `llm_severity` (F3) | the **official severity is that of the LLM** (see [ADR-10](../documentation/decisiones/ARD-10.en.md)); the deterministic from F2 remains as audit in the internal artifact, not here |
| `explanation` | `llm_root_cause` (F3) | explanation in natural language |
| `action` | `llm_recommended_action` (F3) | concrete action with responsible party |

**Block 2 — Support for Phase 4 app (for reading, not recomputing):**

| Group | Columns | Purpose |
|---|---|---|
| Timeline | `PO_DT, STA_DT, APPROVED_DT, TRAILER_ARRIVE_DT, CHECKIN_DT, CHECKOUT_DT, RECPT_DT` | to draw the course of the PO (selector, #102) |
| Aggravating Factors | `HOT_PO_FLAG, is_short_ship` | mark hot PO / short ship in the view |
| Matching | `REASON_DSC, llm_matches_reason` | show if the diagnosis matches human annotation |

Block 1 (5) + Block 2 (11) = the **16 columns of the base contract** (ARD-21). The following 17 expand that contract in two rounds:

**Tier-1 (8 columns, #158)** — enrichment with already computed upstream data, without additional LLM call:

| Columns | Purpose |
|---|---|
| `llm_confidence, VENDOR_NAME, CARRIER_PARTY_NAME, DC_LOC_NAME, delay_days_calc, excess_vendor_hrs, excess_carrier_hrs, excess_dc_hrs` | confidence of the diagnosis, responsible entities, and excess hours by stage (context of the individual view, Diego) |

**Tier-2 (9 columns, #161, PR #174)** — hybrid output of the action call ([ADR-16](../documentation/decisiones/ARD-16.en.md) track 1, opt-in via `--action-call`; without the flag these 9 columns are **empty, not absent** — the contract of 33 is stable with or without it):

| Columns | Purpose |
|---|---|
| `llm_reasoning, llm_hypothesis, llm_hypothesis_evidence, llm_immediate_action, llm_corrective_action, llm_preventive_action, llm_alternate_hypothesis, llm_discriminative_step, llm_hypothesis_confidence` | main hypothesis with evidence and staged plan, alternative hypothesis with its discriminative step, and a second specific confidence of the hypothesis |

**Row scope:** only the **delayed POs** (`delay_days_calc > 0`) — those the LLM explains and those the app offers in the selector. On-time POs do not enter.

**Rule of contract F3→F4:** Phase 4 **reads** `po_output.csv` and nothing else; **does not recompute** the F1/F2 rules nor calls the LLM again. Therefore, the artifact already contains the timeline and aggravating factors: everything the app needs is in the CSV. (An optional demo of “regenerate this PO live” would be the only exception and is handled separately from the main flow.) The contract is bolstered by `tests/test_handoff_f3.py`.

The `po_output.csv` of the deliverable is generated with the official backend, **OpenAI**. Add `--action-call` to also populate the tier-2 (second call, incurs additional API costs per PO):

```bash
python llm_integration.py --mode full --backend openai --action-call
```

## Prompt Design

The prompt is built by `build_prompt()` in `llm_integration.py`: it is the only source of the prompt (ADR-12), interpolates the data from each delayed PO, and prefixes the production few-shot examples. The production configuration is **few-shot C3** (three examples), validated against the quality benchmark; the details of the selection are in [Status of few-shot](#status-of-few-shot). The model card of the system, with model, intended use, and limits, is in [MODEL_CARD.md](MODEL_CARD.en.md).

### What goes into the prompt and why

The prompt injects all the arithmetic already resolved by the previous phases and instructs the model to interpret without recalculating. The blocks, in order:

| Block | Fields | Why it enters |
|---|---|---|
| PO DATA | `PO_NBR`, `VENDOR_NAME`, `DC_LOC_NAME`, `CARRIER_PARTY_NAME` | identify the actors that the action may name |
| TIMELINE | `STA_DT`, `RECPT_DT`, `APPROVED_DT`, `TRAILER_ARRIVE_DT`, `CHECKIN_DT`, `CHECKOUT_DT` | the raw dates; the model reads them for cross-checking, not for recalculating (#91/ADR-14) |
| CALCULATED METRICS | `delay_days_calc`, `yard_wait_calc_hrs`, `dock_calc_hrs`, and the excess per stage (`excess_vendor_hrs`, `excess_carrier_hrs`, `excess_dc_hrs`) | the already resolved arithmetic that the model cites textually; excesses are omitted in Indeterminate, where attribution was intentionally withdrawn (ADR-14) |
| AUTOMATIC CLASSIFICATION | `stage_primary`, `stage_multi` | the deterministic verdict from F2 is the truth source for the stage; the model does not re-decide it |
| ADDITIONAL CONTEXT | `HOT_PO_FLAG`, `_short_ship`, `is_rescheduled`, `indeterminado_substage`, `REASON_DSC` | aggravating factors and meta-signals; rescheduling enters as neutral context, not as aggravating factor (#67); the subcategory only when the stage is Indeterminate (#135) |

After the context, two instruction blocks close the prompt. INSTRUCTIONS prohibits recalculating dates or hours and inventing figures, and demands to cite the provided figures verbatim. HOW TO REASON (#143) specifies that the stage the model names is exactly that of the classification, that the `REASON_DSC` is used for cross-referencing and never to replace the stage, and describes the combinatorial nature of the four stages against the three possible relations with the reason (matches / diverges / empty). This block prevents the model from copying the form of the examples as a template.

### What is requested (JSON contract)

A JSON is requested with five keys, without additional text:

| Key | Content |
|---|---|
| `root_cause` | 2-3 sentences: exact stage, cited delay from the data, relationship with the `REASON_DSC` and aggravating factors |
| `recommended_action` | specific action to the responsible party of the measured stage, anchored in the PO data |
| `severity` | `HIGH` / `MEDIUM` / `LOW`; decided by the LLM (see [MODEL_CARD.md](MODEL_CARD.en.md) and ADR-10) |
| `matches_reason_code` | `true`/`false` according to whether the cause matches the `REASON_DSC` |
| `confidence` | 0.0 to 1.0 |

The severity rules that the prompt delivers (HIGH for hot PO with high delay or short ship; MEDIUM for delay without aggravating factors; LOW for borderline) interpolate the thresholds from `rules_config.json` (#121), the same unique source used by F2, instead of fixed values in the code.

### How the response is parsed

Parsing is centralized in `_parse_llm_json()` to avoid duplicating logic between backends. It looks for the first block `{...}` in the response, loads it, and normalizes the five keys (accepts English variants: `root_cause`, `severity`, etc.). If the model does not return usable JSON and `fallback=True`, it constructs an emergency dict with the trimmed raw text; this fallback is meant for the `local` backend (Qwen), which may respond in free text. Cloud backends (`openai`, `claude`, `deepseek`) expect valid JSON and do not use the fallback.

### Status of few-shot

Production uses **few-shot C3**: three examples, one for each attributable stage (Vendor, Carrier, DC), selected deterministically with `select_examples(3, stages=["Vendor","Carrier","DC"])` from the audited pool `fewshot_pool.json`. The examples come from real mismatches of F2 (the computed classification diverges from the human `REASON_DSC`), disjoint from the 20 PO benchmark to avoid contaminating the metric; each example mirrors the form of the prompt but curated in content, and does not include the timeline of dates (showing it would re-teach to recalculate).

C3 won the quality benchmark (`eval_quality_20pos.en.md`) and was re-validated at production temperature: `fixtures/eval_quality_20pos_C3_t09.md` reports 20/20 with human validation at temperature 0.9, without regression against the anchor 0.3. The `--zero-shot` flag disables the few-shot and reproduces the baseline C0 for comparisons. The pool retains two synthetic examples of Indeterminate (for future dynamic selection work); C3 does not select them, and the Indeterminate stage is resolved by the HOW TO REASON block.

### Second call: action plan (tier-2)

Under the `--action-call` flag (ARD-16), each PO receives a second call with `build_action_prompt()`: a planner role with decision-making authority that produces a hybrid contract (reasoning → main hypothesis with its plan → alternative hypothesis with its discriminative step → confidence) and goes through a quality control by rules. The first call diagnoses; this one plans the mechanism under the stage level. The detail lives in ARD-16.

### Prompt iterations

| Iteration | Issue / ADR | What changed |
|---|---|---|
| Base zero-shot | #91 / #67 | pre-resolved arithmetic, interpret without recalculating, rescheduling as neutral context |
| Few-shot | #99 / ADR-12 | examples that teach reasoning; C3 (three examples) wins the benchmark |
| Anti-overfitting | #143 / ADR-14 | HOW TO REASON block: authority of the stage over the `REASON_DSC`, combinatorial nature of the four stages |
| Externalized threshold | #121 | severity thresholds interpolated from `rules_config.json` (unique source with F2) |
| Indeterminate subcategory | #135 | `indeterminado_substage` (sin_datos / sin_causa_dominante) in the prompt |
| Domain context | #151 | opt-in block by (actor × signal), off by default |
| Validated temperature | ADR-13 | C3 re-validated at 0.9 (`_C3_t09`, 20/20), without regression vs 0.3 |

## Error handling

- **401/403 (Invalid API key or lack of permissions):** the script fails immediately without retrying and displays a clear message indicating which environment variable to check.
- **Other HTTP errors (429, 500, etc.):** it retries up to 3 times with a 2-second pause between attempts.
- **Network errors (timeout, connection):** same retry behavior.
- If a PO fails after retries, it is marked as `FAILED` in the progress bar and continues with the next one (does not halt the entire batch).

## Troubleshooting

**`ModuleNotFoundError: No module named 'dotenv'`**
The `.venv` is not activated. Run `source .venv/bin/activate` (Windows: `.venv\Scripts\activate`) from the root of the repo before executing the script.

**`HTTP Error 401: ... api key ... is invalid`**
The key in `.env` is not valid or has extra spaces/characters copied. Verify it directly with a test `curl` to the provider's endpoint, or regenerate the key from its dashboard.

**`CSV not found in ...`**
The script expects the raw CSV in `../data/raw/po_root_cause_synthetic.csv` relative to the root of the repo. Check that the file exists at that exact path.

## Design Notes

- **Prompt guideline (#91).** `build_prompt()` follows the mentor's template (kickoff §03 / README §5): all arithmetic arrives already resolved (timeline + CALCULATED METRICS) and the model only **interprets**. The prompt explicitly prohibits recalculating dates/times or inventing figures, and demands **to cite the given figures verbatim** (e.g., "a delay of 4.2 days") to ensure the explanation is auditable against the data. The requested explanation (`root_cause`) consists of **2-3 sentences** with the mentor's elements: exact stage of the delay (Vendor/Carrier/DC), quantified delay cited, match with `REASON_DSC`, and aggravating factors (hot PO, short ship); the action must be operable and name the responsible party, not generic. The JSON output contract (5 keys) did not change. Design decisions: citation of the figure is required (auditability); short ship remains as a boolean "Yes/No" (the `{short_ship_pct}%` from the mentor's template remains as a follow-up). The severity threshold continues at `> 3 days` (ADR-10); externalizing it to `rules_config.json` is work for #121.
- **Reschedule citation as context (#67).** The prompt includes `is_rescheduled` (from F2) as **neutral context** ("Was the delivery appointment rescheduled? Yes/No"), not as a stage or automatic aggravating factor. Reason: the mentor (06-16) discarded rescheduling as *vendor signal* because it describes an event, not the cause; including it as an aggravating factor would reintroduce the pro-vendor bias that the closure of F2 corrected. The LLM uses it to better judge the match with `REASON_DSC` (many human reason codes are "Rescheduled by vendor"), without forcing the stage to Vendor for it — the stage is still marked `stage_primary`.
- **Unique source of the prompt (#99 / ADR-12).** The prompt sent to the model is built by `build_prompt()` in `llm_integration.py`: it is the **only** source of the prompt, interpolates the data from each PO and optionally prefixes few-shot examples (parameter `examples`). The old draft `prompt_template.txt` was **removed**: it was not loaded from the code and had diverged (taxonomy of 6 stages that are not the 4 states of F2, instructions that invited recalculation against guideline #91, and a severity threshold `> 7 days` contradictory to ADR-10). Keeping it risked it being reused without context and reversing current decisions. The design of the prompt (few-shot that teaches reasoning, unique source) is recorded in ADR-12.
- **Validated few-shot (#99).** Against the 20 POs benchmark (`eval_quality_20pos.en.md`, seed 42), zero-shot and three few-shot combinations with examples from the audited pool (`fewshot_pool.json`, stratified: the strongest mismatch from each stage) were compared. **C3** (3 examples) won: verdict **19/20 (4.75/5)** versus 13/20 from the zero-shot, exceeding the mentor's target (4/5), without degrading (a)/(b). The few-shot addresses the deficit of (c): it converts generic actions into specific actions that cite the measured signal and direct to the correct responsible party. **C3 is the production configuration**: `main()` populates `examples` with `select_examples(3, stages=["Vendor","Carrier","DC"])`, `add_llm_explanations` passes it to `build_prompt`, and the `po_output.csv` of the deliverable is generated with C3. The 4.75/5 is the benchmark that selected C3 against C1/C2, not the final figure: the hardening of the prompt in #143 closed the only remaining failure (still at temp 0.3), and the re-validation at the actual production temperature in `fixtures/eval_quality_20pos_C3_t09.md` confirms **20/20 (5/5)** at 0.9, with no regression. **The headline figure of the deliverable is 5/5** — the one that describes the configuration that produces `po_output.csv` today; see [documentation/metricas-proyecto.md](../documentation/metricas-proyecto.en.md) for the complete progression with provenance. The `--zero-shot` flag reproduces the baseline C0. See the section [Prompt Design](#prompt-design).
- The parsing of the LLM's JSON response is centralized in `_parse_llm_json()` to avoid duplicating logic between backends.
- The `local` backend (Qwen) has fallback to free text if the model does not return valid JSON; cloud backends (`claude`, `deepseek`) do not, as they are expected to adhere to formatting instructions more reliably.
- The three backends share the same interface (`call(prompt, max_retries)`), so adding a new provider only requires implementing a new class and registering it in `create_backend()`.
