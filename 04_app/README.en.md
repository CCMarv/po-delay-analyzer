# Phase 4 — Demo + Final Evaluation (app)

> Phase document. For now **only in Spanish**; the English version will be added at the
> end of the development.

Phase 4 presents the results of the analysis to two user profiles. It does not produce
new analysis: **it reads** the artifact from Phase 3 and displays it. The app is Streamlit; all
the cleaning logic (F1), classification (F2), and explanation (F3) have already occurred upstream.

## 1. Input: the contract F3→F4 (#100)

The primary input of the app is the contract F3→F4:

```
data/processed/po_output.csv
```

It is generated by Phase 3 (`03_llm_integration/llm_integration.py`, `export_deliverable_csv`).
The app **does not recompute** Phase 1/2 nor does it call the LLM again: it only reads artifacts already
produced upstream. The CSV contract is protected by `tests/test_handoff_f3.py`.

The aggregated view (§2) also consumes scorecards by entity and the executive synthesis of the network
(`agente1_raw.txt`): indicators and narrative derived that are generated offline/via LLM (§3), do not
form part of the mentor contract, are regenerable, and are outside version control. `agente1_raw.txt` is a real dependency of Phase 3→Phase 4, not an isolated artifact ([ARD-21](../documentation/decisiones/ARD-21.en.md)).

CSV structure (the mentor's five first, in order; then support for the app):

| Block | Columns | Purpose |
|---|---|---|
| Mentor Contract | `PO_NBR, stage, severity, explanation, action` | Identity + diagnosis (`severity`, `explanation`, `action` outputted by the LLM — see ADR-10) |
| Timeline | `PO_DT, STA_DT, APPROVED_DT, TRAILER_ARRIVE_DT, CHECKIN_DT, CHECKOUT_DT, RECPT_DT` | Draw the PO journey |
| Aggravating Factors | `HOT_PO_FLAG, is_short_ship` | Mark urgency / incomplete shipment |
| Concordance | `REASON_DSC, llm_coincide_con_reason` | Show if the diagnosis matches the human annotation |
| Tier 1 Enrichment | `llm_confianza, VENDOR_NAME, CARRIER_PARTY_NAME, DC_LOC_NAME, delay_days_calc, excess_vendor_hrs, excess_carrier_hrs, excess_dc_hrs` | Confidence of the diagnosis, responsible entities, and excess hours by stage (context of the individual view) |
| Tier 2 Differential Diagnosis | `llm_razonamiento, llm_hipotesis, llm_hipotesis_evidencia, llm_accion_inmediata, llm_accion_correctiva, llm_accion_preventiva, llm_hipotesis_alt, llm_paso_discriminante, llm_confianza_hipotesis` | Hybrid output from ARD-16: main/alternative hypotheses, evidence, discriminative step, staggered plan, and 2nd confidence |

Row scope: only late POs (`delay_days_calc > 0`).

If a view needs a data point that the CSV does not provide, the contract is expanded (#100) in Phase 3,
**not** recomputed in the app.

## 2. Output: two views, two people (ADR-09)

The design is organized by **mode of consumption**, not by entity in the chain:

- **Individual View — person Diego (#102).** Query of a PO: timeline, responsible stage,
  explanation and action from the LLM, severity, and concordance with the REASON_DSC.
  Consumes the prose of the LLM.
- **Aggregated View — person Ravi (#103).** Report by batch: metrics, distribution by stage,
  drill-down. Consumes aggregates, not prose.

Details of the personas in `../documentation/user_personas.en.md`.

Streamlit is not the only channel: `telegram_bot/` ([ADR-20](../documentation/decisiones/ARD-20.en.md))
exposes a second front-end with fixed commands for each person (`/po`, `/timeline`, `/alertas`,
`/hot` for Diego; `/kpi`, `/scorecards`, `/distribucion`, `/mismatches` for Ravi) on the same contract F3→F4 — without free reasoning or LLM calls at query time, different from the deferred conversational chatbot (#160).

## 3. How to Run

```bash
# 1. Generate the input artifact (Phase 3), if it does not exist:
#    produces data/processed/po_output.csv
#    --action-call (opt-in) populates the columns of tier 2 differential diagnosis
#    (#161/#174); without that flag those columns will be empty.
python 03_llm_integration/llm_integration.py --backend openai --action-call   # or another backend

# 2. Generate scorecards by entity (offline, without API):
#    produces data/processed/scorecards/reporte_{vendors,carriers,dcs}.json
python 03_llm_integration/scorecard_core.py \
    data/processed/df_classified.csv data/processed/scorecards
#    On Windows, if the console fails with emojis, prefix PYTHONUTF8=1

# 3. Generate the executive synthesis of the network (GASTA API, ADR-19), input of Network
#    Intelligence: produces data/processed/agente1_raw.txt. Must run after
#    step 2 (reads those scorecards). Requires --actor all to consolidate the report.
python 03_llm_integration/llm_integration_network_intelligence_view.py --actor all

# 4. Launch the app:
streamlit run 04_app/app.py
```

## 4. Status

The app reads three artifacts, all regenerable and outside version control:

- `data/processed/po_output.csv` — base contract + tier-1/tier-2 (§1, ARD-21), primary input
  for both views.
- `data/processed/scorecards/reporte_{vendors,carriers,dcs}.json` — indicators by
  entity that feed the aggregated view. They are produced offline
  `03_llm_integration/scorecard_core.py` on `df_classified.csv`; the app reads them, does not
  recompute the statistical layer, nor calls any API.
- `data/processed/agente1_raw.txt` — executive synthesis of the network by actor. It is produced
  by `03_llm_integration/llm_integration_network_intelligence_view.py --actor all` (ADR-19,
  uses API) from the scorecards; it is consumed by Network Intelligence, not the app in
  general.

The two views, individual (#163) and aggregated (#164), are rebuilt on the design system of the phase. Included in the quality elevation deployment batch:

- Tier 2 differential diagnosis panel in the individual view (#175/#176): main/alternative hypotheses, discriminative step, and staggered plan.
- Light/dark theme adapted to Plotly graphs (#186, ARD-17).
- Correction of navigation from the landing page to the views (#187).
